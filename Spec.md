# 要件
LLM を活用して、ユーザーが手元の Excel データを構造化 DB に変換し、自然言語で検索・集計できるデータ管理・分析ツールを作る。

## プロダクト概要
- ユーザーは複数の Excel ファイルをアップロードできる。
- システムは LLM を用いて各ファイルの構造を分析し、列の意味やデータ型を推定する。
- 表記ゆれ、不要な空白、似た意味を持つ列名の違いなどを吸収し、DB 化しやすい形へ整える。
- DB 化したデータに対して、ユーザーは自然言語で欲しい情報を問い合わせできる。
- LLM は問い合わせを SQL に変換し、実行結果を返す。

## 最小機能

### 1. Excel 取り込み
- 複数の Excel ファイルを入力対象とする。
- LLM が列名、値の傾向、データ型、列同士の意味的な近さを分析する。
- 例えば以下を対象にする。
  - 同義だが名前が異なる列の統合候補提示
  - 不要な前後スペースのクレンジング
  - 基本的な表記ゆれの吸収

### 2. DB 作成前の確認フロー
- DB を作る前に、LLM は統合方針をユーザーに提案する。
- 提案例:
  - `X.xlsx` の `A列` と `Y.xlsx` の `B列` は同じ概念とみなし統合候補
  - ただし別概念の可能性がある列は統合しない候補として保持
- ユーザーは提案内容を見て、承認するか、自然言語で補足指示を返す。
- 指示例:
  - `X.xlsx のA列と Y.xlsx のB列は似ていますが、業務上は別概念なので統合しないでください`
- このやり取りは、作成後の修正ではなく、DB 作成前の合意形成として扱う。
- ユーザーが合意した内容をもとに DB を生成する。

### 3. データ保持方針
- 元の Excel に近いデータも保持する。
- あわせて、LLM の提案とユーザー承認を反映した統合後データも保持する。
- これにより、元データとの対応関係を追いやすくする。

### 4. 自然言語検索
- ユーザーは自然言語でデータ参照を依頼できる。
- 例:
  - `XXX に関する YYY のデータを見たい`
  - `部署別の売上合計を見たい`
- LLM は問い合わせ内容を SQL に変換し、DB に対して実行する。
- 対象は参照系と集計系に限定する。
- `INSERT` `UPDATE` `DELETE` などの更新系は対象外とする。

### 5. 検索結果の表示
- 検索結果として以下を表示する。
  - 結果テーブル
  - 生成された SQL
  - 結果の自然言語説明
- ユーザーが、結果が意図通りかを確認できることを重視する。

## ユーザー入出力

### 入力
- 複数 Excel ファイル
- DB 作成前の確認時に与える自然言語フィードバック
- データ検索時の自然言語クエリ

### 出力
- 列統合候補とスキーマ案
- 承認後に生成された DB
- SQL 実行結果
- 生成 SQL
- 実行結果の説明文
